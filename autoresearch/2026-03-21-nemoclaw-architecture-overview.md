# Nemoclaw Architecture: Fully Privatized Personal AI System

## The Architecture

Nemoclaw is the orchestration layer that connects locally-trained models, an Obsidian knowledge vault, and Claude Code into a single privatized AI system. No data leaves the local network. All models run on owned hardware. Training happens locally or on user-controlled cloud instances.

```
NEMOCLAW (orchestrator)
  │
  ├── MAC MINI M4 16GB (always-on, 3-5W idle, $5/year electricity)
  │   │
  │   ├── SmolLM2-135M micro (724MB RAM)
  │   │   • Routes queries between models
  │   │   • Classifies intent, tags notes, extracts entities
  │   │   • Retrieves vault context for other models
  │   │   • LoRA retrained daily on vault data (15 min, $0)
  │   │
  │   ├── Phi-4-mini 3.8B nano (3GB RAM, Q4)
  │   │   • Synthesizes vault content into explanations
  │   │   • Writes session summaries to vault
  │   │   • Reviews plans for over-engineering
  │   │   • Generates training data for micro
  │   │   • No fine-tuning needed — uses RAG for personalization
  │   │
  │   ├── nomic-embed-text (300MB RAM)
  │   │   • Semantic search over entire vault
  │   │   • Auto-updates when files change
  │   │
  │   └── OBSIDIAN VAULT (primary copy)
  │       • Shared memory bus for all models
  │       • Synced to PC via Syncthing (free, P2P, local network)
  │       • Three access layers: embeddings, MCP read/write, graph RAG
  │
  ├── PC 64GB RAM, NO GPU (on-demand)
  │   │
  │   ├── 14B model via Ollama (12GB RAM, Q4)
  │   │   • Link-to-learning: URL → personalized guide
  │   │   • Comprehends docs, maps to user's workflow, teaches
  │   │   • CPU inference at 2-6 tok/s (batch, not chat)
  │   │   • LoRA fine-tuned monthly on RunPod ($1-6/session)
  │   │
  │   └── OBSIDIAN VAULT (synced copy)
  │       • Receives guides generated by 14B
  │       • Syncthing propagates back to Mac Mini
  │
  └── CLAUDE CODE (external, API — the only non-local component)
      • User's primary development tool
      • Hooks (SessionStart, Stop, PreCompact, SessionEnd) → sidecar
      • Micro captures decisions/preferences from sessions
      • Nano injects context briefings into CLAUDE.md
      • Nemoclaw controls what context flows to Claude
```

## Data Flow

**Persistent memory loop (always-on, Mac Mini):**
```
Claude Code session → hooks fire → micro classifies exchanges →
nano synthesizes memories → writes to vault → embeddings update →
next session starts → micro retrieves context → nano compresses
briefing → injects into CLAUDE.md → Claude reads it automatically
```

**Link-to-learning pipeline (on-demand, PC):**
```
User sends URL → scraper fetches content → micro searches vault
for related context → sends content + context to PC's 14B via
Ollama API → 14B generates personalized guide → saves to vault →
Syncthing syncs to Mac Mini → embeddings index updates →
guide is searchable and retrievable for future sessions
```

**Daily micro retraining (automated, Mac Mini):**
```
Nano reads day's changed notes → generates instruction JSONL →
LoRA fine-tune SmolLM2-135M via MLX (15 min) → swap 3MB adapter →
micro's routing/classification improves → no restart needed
```

## Privacy Guarantees

| Component | Where data lives | Who controls it |
|-----------|-----------------|-----------------|
| Obsidian vault | Mac Mini + PC (local) | You |
| Model weights | Mac Mini + PC (local) | You |
| LoRA adapters | Mac Mini (local) | You |
| Embeddings index | Mac Mini (local) | You |
| Training data | Mac Mini (local) | You |
| Vault sync | Syncthing (P2P, no cloud) | You |
| 14B fine-tuning | RunPod (ephemeral, your instance) | You |
| Claude Code context | Anthropic API (only what Nemoclaw sends) | You choose what flows |

The only data that leaves your network is what Nemoclaw explicitly sends to Claude Code via hooks. Everything else — models, training, memory, sync — runs on hardware you own, on your local network.
