## Research Brief

**Goal**: What are the optimal model sizes, hardware assignments, and software stack choices for building a 3-node personal AI swarm (super/nano/micro) that serves as a persistent memory sidecar for Claude Code, integrates with an Obsidian vault, and helps a non-developer right-size project plans before implementation?

**Source scope**: Web (official docs only — Ollama, llama.cpp, MLX, Obsidian, model cards from HuggingFace/official repos) + codebase (`use-case-examples/`, `reference/`)

**Iterations**: 5

**Quality bar**: Standard

**Threads to explore**:
1. **Hardware matrix** — Mac Mini M4 (16GB) vs M4 Pro/Max (48GB+) vs PC (64GB RAM + RTX GPU): what runs where, inference speed benchmarks on Apple Silicon vs CUDA, memory requirements per model size, and a buy recommendation. User is willing to upgrade hardware later as needed.
2. **Model selection by tier** — Super: the biggest/newest Nemotron model available (or equivalent flagship), user is OK upgrading hardware to run it. Research what hardware is needed for Nemotron Super (49B), Nemotron Ultra, or latest equivalent. Nano (1B-4B range): specific model recommendations with focus on coding assistance, knowledge management, plan review. Compare Phi-3/3.5, Qwen2.5, Gemma 2, Mistral, Llama 3.x at relevant sizes. Micro: ~50M params — research what's available at this size or what would need to be distilled/trained. This is the persistent sidecar so it needs to be fast and always-on, handling classification, routing, context retrieval, and simple Q&A against the Obsidian vault
3. **Obsidian vault integration** — How to wire local models to read/write an Obsidian vault as persistent memory. Obsidian MCP server, local plugins, file-based approaches. The vault is the knowledge base — models need to search it, update it, and use it as context
4. **Claude Code sidecar architecture** — How the micro/nano acts as persistent memory across Claude Code sessions. MCP servers, CLAUDE.md injection, session handoff patterns. The sidecar should remember project context, decisions, and progress that Claude Code forgets between sessions
5. **Orchestration and wiring** — Practical stack for connecting the 3 nodes: Ollama, Open WebUI, LiteLLM, or custom. How the super delegates to nano/micro. How the user interacts (single entry point vs multiple interfaces)

DO NOT re-derive these concepts — build on them with practical specifics.

**Suggested starting point**: Web search for current Apple Silicon (M4 Pro/Max) inference benchmarks with Ollama/MLX for models in the 1B-14B range. This anchors the hardware decision which cascades into everything else.

**User context**:
- Non-developer, learns fast, tends to over-engineer
- Already uses Claude Code as primary dev tool
- Already has Obsidian vault as knowledge base
- Willing to buy new hardware (Mac Mini M4 Pro/Max 48GB+)
- Primary use case: persistent memory sidecar that ingests new AI developments, maps them to personal projects, and reviews spec/plans before implementation
